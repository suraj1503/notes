What is PostgreSQL?
     - PostgreSQL is an open-source relational database known for strong ACID compliance, advanced indexing, concurrency handling through MVCC, and support for both relational and semi-structured JSON data. 
     - It is widely used in scalable backend systems.

Difference between SQL and NoSQL databases?
     - SQL databases are relational, use tables, and have strict schemas
     - NoSQL databases are non-relational, use flexible schemas, and are good for large, unstructured datasets

Why Companies Use PostgreSQL
    - ACID Compliance
    - Ensures: consistency, rollback support, safe transactions
    - MVCC (Multi-Version Concurrency Control):
        - MVCC is the core mechanism used by PostgreSQL to handle concurrent reads and writes without locking rows for reads.
        - Multiple transactions can read and write data simultaneously without blocking each other.
        - Improves concurrency and performance.
        - PostgreSQL creates multiple versions of rows.
        - Old row version remains.
        - New row version is created.
        - Each transaction sees a consistent snapshot of data.
        - use "VACCUM" to clean up dead tuples (old versions).
    - Powerful Indexing:
        - B-tree, Hash, GIN, GiST, BRIN

PostgreSQL Architecture
    - Clients send SQL queries
    - SQL Parser validates syntax
    - Query Planner creates execution plan
    - Executor runs plan
    - Data stored in Tables (Disk Files)
    - Shared Buffers :  
        - This is PostgreSQL's main memory cache.
        - When data is read from or written to the database, it passes through the Shared Buffers.
        - And If data does not exists in Shared Buffer, it will load from disk to Shared Buffer and then process.
        - Frequently accessed data stays in memory for faster access.
    - Indexes speed up reads
    - WAL (Write-Ahead Log) ensures durability

Joins
    - INNER JOIN (return only matching rows from both tables)
        - Eg. 
            SELECT users.name, orders.amount
            FROM users
            INNER JOIN orders
            ON users.id = orders.user_id; 
    - LEFT JOIN (return all rows from left table and matching rows from right table)
        - Eg. 
            SELECT users.name, orders.amount
            FROM users
            LEFT JOIN orders
            ON users.id = orders.user_id;
    - RIGHT JOIN (return all rows from right table and matching rows from left table)
        - Eg. 
            SELECT users.name, orders.amount
            FROM users
            RIGHT JOIN orders
            ON users.id = orders.user_id;
    - FULL OUTER JOIN (return all rows from both tables)
        - Eg. 
            SELECT users.name, orders.amount
            FROM users
            FULL OUTER JOIN orders
            ON users.id = orders.user_id;
    - CROSS JOIN (return all possible combinations of rows from both tables)
        - Eg. 
            SELECT users.name, orders.amount
            FROM users
            CROSS JOIN orders;
    - SELF JOIN (join a table with itself)
        - Eg. 
            SELECT users.name, orders.amount
            FROM users
            JOIN users u2
            ON users.id = u2.id;
    - NATURAL JOIN (join based on common column names)
        - Eg. 
            SELECT users.name, orders.amount
            FROM users
            NATURAL JOIN orders;

Schema vs Database:
    - A Schema is a namespace that organizes database objects such as tables, views, and functions. It allows you to group related objects together.
    - A Database is a collection of schemas and their objects.

Transactions:
    - A database transaction is a single unit of work that consists of one or more database operations.
    - A transaction is either fully completed or fully rolled back, ensuring data consistency.
    - ACID Properties:
          - Atomicity:
               - All operations in a transaction are completed successfully, or none of them are.
          - Consistency:
               - A transaction brings the database from one valid state to another valid state.
          - Isolation:
               - Concurrent transactions do not interfere with each other.
          - Durability:
               - Once a transaction is committed, its changes are permanent and survive system failures.
        - Eg.
            BEGIN;

            UPDATE accounts
            SET balance = balance - 100
            WHERE id = 1;

            UPDATE accounts
            SET balance = balance + 100
            WHERE id = 2;

            COMMIT;

            ELSE
                ROLLBACK;
    - Isolation Levels: 
        - Read Uncommitted - Allows reading uncommitted data.
        - Read Committed (Default) - Allows reading committed data only.
        - Repeatable Read - Allows reading committed data and ensures that the same row will return the same value if read multiple times within a transaction.
        - Serializable - Prevents all concurrency issues by executing transactions serially.

Locks & Concurrency:
- Postgres uses: MVCC (Multi-Version Concurrency Control)
    - PostgreSQL uses MVCC for high concurrency by allowing readers and writers to work without blocking each other
    - while row-level locks like FOR UPDATE are used to prevent race conditions during concurrent writes
    - and deadlocks are automatically detected and resolved by aborting one transaction.

- Allows:
   - readers not blocking writers
   - writers not blocking readers
- Improves:
   - concurrency
   - performance
- PostgreSQL creates:
   - multiple versions of rows
   - each transaction sees consistent snapshot
- Row-Level Lock:
   - SELECT * FROM users WHERE id = 1 FOR UPDATE;
   - locks selected row
   - prevents concurrent updates
- Used in:
   - wallet systems
   - inventory systems
   - booking systems
- Race Condition:
   - multiple transactions update same data simultaneously
   - can cause inconsistent data
- Deadlock:
   - transactions wait on each other
   - PostgreSQL detects and aborts one transaction
- Best Practices:
   - keep transactions short
   - access rows in same order
   - avoid long-running locks
- Isolation Levels:
   - Read Committed (default)
   - Repeatable Read
   - Serializable

WAL (Write-Ahead Logging):
    - WAL ensures data durability by logging all changes to disk before applying them to the database.

Normalization:
    - Database design technique to reduce data duplication and inconsistency
    - Organizes data into multiple related tables
    - Improves:
        - consistency
        - storage efficiency
        - maintainability
    - Example:
        - instead of repeating user_name in orders table
        - create:
            - users table
            - orders table
    - Use Foreign Keys:
        - orders.user_id REFERENCES users(id)
        - Prevents:
        - update anomaly
        - insert anomaly
        - delete anomaly
    - Benefits:
        - less duplicate data
        - better data integrity
        - easier updates
    - Normal Forms:
        - 1NF (First Normal Form):
            - each column should contain atomic/single values
            - no multiple values in one column
            - Example:
                - Bad:
                    - phones = "123,456"
                - Good:
                    - separate rows for each phone
        - 2NF (Second Normal Form):
            - table must already be in 1NF
            - no partial dependency on composite primary key
            - non-key columns should depend on full primary key
            - Example:
                - if (student_id, course_id) is composite key
                - course_name should depend on course_id only
                - move course data to separate table
        - 3NF (Third Normal Form):
            - table must already be in 2NF
            - no transitive dependency
            - non-key column should not depend on another non-key column
            - Example:
                - city -> state
                - store city/state in separate table
    - Drawback:
        - too many joins can slow reads
    - Denormalization:
        - intentionally duplicate data for faster reads
