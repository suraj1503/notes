VM
    its a virtual machine to run multiple application on the same machine
    the software used for this called hyperwiser (vmWare, VirtualBox)
    you can run multiple VMs in parallel in this architecture.

Problems with Traditional Virtualization
     Very Heavy - Each VM contains a full OS
     Slow Startup - Starting a VM means booting an entire OS. [VM boot time = 1–2 minutes]
     High Resource Usage
     Deployment Issues - "It works on my machine"

Docker
     - Docker solves these problems using containers instead of virtual machines.
     - Docker is written in Go and runs natively on Linux.
     - Instead of creating multiple OS, Docker shares the host OS kernel.
     - Your NodeJS runtime does NOT come from the host machine. it came from Image runtime
     - Why Containers Are Still Lightweight
          * They reuse image layers
          * They use minimal Linux distributions
     - Why Docker Is Fast
          * Share the same OS kernel
          * No OS boot required
          * Lightweight processes

Docker terminologies
     - Container:
          * A container is a running isolated environment for your application.
          * It contains: Your code, Runtime, Libraries, Dependencies, Environment configs
          * Example
               Suppose you built:
                    * Node.js backend
                    * MongoDB
                    * Redis
     - Image
          - A Docker image is a blueprint/template used to create containers
          - Container = Running instance
          - Image = Read-only template  
          - Example: 
               - Image: Node.js + Express App + Dependencies
               - Container: Running Node.js application 
               - docker pull node:20  (base image)
               - docker build -t my-app .
               - docker run -p 3000:3000 my-app
               
     - Dockerfile
          - A Dockerfile is a text file that contains instructions for building a Docker image.
          - A Docker image is not one single file. It is built as a stack of layers.
          - Each instruction in a Dockerfile creates a new layer
          - Dockerfile Instructions
               - FROM        → Base image
               - WORKDIR     → Create working directory
               - COPY        → Copy files
               - RUN         → Execute commands
               - CMD         → Start application
          - Example
               - Dockerfile
                    FROM node:20
                    WORKDIR /app
                    COPY package.json ./
                    RUN npm install
                    COPY . .
                    EXPOSE 3000
                    CMD ["npm", "start"]
               - Bad Dockerile:
                    FROM node:20
                    WORKDIR /app
                    COPY . .
                    RUN npm install
                    CMD ["node","server.js"]
               - Note: Above example installs dependecny and file change, but only file is changed not dependency.
          - Docker caches the layers
          - How Docker Decides Whether to Use Cache
               - The instruction in the Dockerfile
               - The files used by that instruction (Host ka file change hua toh)
          - Flow
               - If both are unchanged → Docker uses the cached layer.
               - If something changes → Docker rebuilds that layer and all layers after it.
          - Example
               Layer 1 → Base Image (node:20)
               Layer 2 → WORKDIR /app
               Layer 3 → COPY package.json
               Layer 4 → RUN npm install
               Layer 5 → COPY . .
               Layer 6 → CMD
        
     - Docker Engine:
          - Docker Engine is the core runtime that: Builds images, Runs containers, Manages networking, Manages storage. It consists of three main components:
               - Docker Daemon (dockerd): The background service that manages Docker objects (images, containers, volumes, networks).
               - Docker CLI (Docker Client): The command-line tool used to interact with the Docker daemon.
               - Docker Compose: A tool for defining and running multi-container Docker applications.

     - Docker Hub:
          - Docker Hub is a public registry for Docker images (like AppStore for Applications or GitHub for code).
          - It is a cloud-based service that allows you to store and share your Docker images.
          - Example: 
               - docker pull node:20 (pulls image from Docker Hub)
               - docker push my-app:latest (pushes image to Docker Hub)
     
     - Docker Registry:
          - A storage system for Docker images.
          - Docker Hub is one type of public registry
          - You can also host your own private registry
          - Example: 
               - docker pull node:20 (pulls image from Docker Hub)
               - docker push my-app:latest (pushes image to Docker Hub)

     - Base Image:
          - A base image is a template used to create Docker images.
          - It contains the OS and the runtime environment for the application.
          - Example: 
               - FROM node:20 (base image)
               - FROM alpine (base image)

     - Layers:
          - A Docker image is not one single file. It is built as a stack of layers.
          - Each instruction in a Dockerfile creates a new layer.
          - When you run a container from an image, Docker creates a thin writable layer on top of the image layers.
     
     - Volume:
          - Docker volumes are used to persist data generated by and used by Docker containers.
          - Because containers are temporary.
          -If container dies: Internal data disappears. Volumes solve this.
     
     - Bind Mount:
          - Maps local machine directory into container.
          - Changes in container reflect on host and vice versa.
          - Example:
               docker run -v /host/path:/container/path my-app
     
     - Port Mapping:
          - Maps container port to machine port.
          - Example:
               docker run -p 3000:3000 my-app
     
     - Network:
          - Docker networks allow containers to communicate with each other.
          - Example:
               docker network create my-network
               docker run -d --network my-network --name db mongo
               docker run -d --network my-network --name app app
     
     - Bridge Network:
          - A virtual network that allows containers to communicate with each other.
          - Example:
               docker network create my-network
               docker run -d --network my-network --name db mongo
               docker run -d --network my-network --name app app
     
     - Host Network:
          - Connect container directly to host's network interface.
          - Faster but less isolation.
          - Example:
               docker run -d --network host my-app
     
     - Docker Compose:
          - Docker Compose is a tool for defining and running multi-container Docker applications.
          - Example:
               docker-compose up
               docker-compose down
     
     - Docker Swarm:
          - Docker Swarm is a container orchestration tool for Docker.
          - Example:
               docker swarm init
               docker swarm join
          
     - Container Lifecycle:
          - Created: Created but not running
          - Running: Running
          - Paused: Temporarily stopped
          - Restarting: Restarting
          - Exited: Stopped
          - Dead: Unresponsive
          
     - Environment Variables:
          - Environment variables are used to configure the container at runtime.
          - Example:
               docker run -e DB_HOST=localhost -e DB_PORT=27017 my-app
     
     -Docker Build:
          - Docker build is used to build Docker images.
          - Example:
               docker build -t my-app .

     - Docker Run:
          - creates and starts a container from an image 
          Eg: docker run myapp
     
     - Docker Exec:
          - runs a command inside a running container.
          Eg: docker exec -it myapp bash 

commands to create a basic docker app:
     - docker build -t my-app . (here . represent current directory from where we run the command) (it creates image)
     - docker images (list all the images)
     - docker run -p 3000:3000 my-app     (myapp docker image u want to run)  ( -p represent port mapping in host and container  Eg:-host port 3000 to container port 3000 , use can change port in host to avoid conflict )
     
     -p (The Bridge)
          - Full form: --publish 
          - Purpose: This maps a port on your computer to a port inside the Docker container.
          - Without this, your app might be running inside the container, but you wouldn't be able to see it in your browser because the container is "locked" from the outside world.

     - 3000:3000 (The Mapping)
          - This is often confusing, but think of it as "Host : Container".
          - First 3000 (Host): This is the port on your laptop. 
          - When you type localhost:3000 in Chrome, it hits this port.
          -Second 3000 (Container): This is the port the app is listening on inside the container (defined by EXPOSE 3000 in your Dockerfile).

     -t (The Tag)
          - Full form: --tag 
          - Purpose: This is like giving your Docker image a nickname or version number. Without this, Docker assigns a random ID. Using a tag makes it easy to remember and run specific versions of your image.
          - Example:
               docker build -t my-app:1.0 .
               docker run -p 3000:3000 my-app:1.0
               docker run -p 3000:3000 my-app

what is K8:
     - Kubernetes is a container orchestration platform that automates deployment, scaling, and management of containerized applications.
     - Operating system for your backend infrastructure
     - You tell it:
          - Run 3 instances of my Node app
          - If one crashes, restart it
          - If traffic increases, scale it

A Pod = running container

Deployment (how many pods you want)

Service (how users access your app)
     - A Service gives:
          - stable URL
          - load balancing across pods
     - A Service in Kubernetes provides a stable network abstraction over a group of pods, enabling consistent access via a fixed IP or DNS name while load balancing traffic across dynamically changing pod instances.”

Basic yaml: 
     apiVersion: apps/v1
     kind: Deployment
     metadata:
          name: node-app
     spec:
          replicas: 3
          template:
               spec:
                    containers:
                         - name: node-app
                           image: my-node-app

Service yaml

     apiVersion: v1
     kind: Service
     metadata:
       name: node-service
     spec:
          selector:
               app: node-app
          ports:
               - port: 80
            targetPort: 3000
     type: NodePort



Key Features (Say these confidently)
     - Auto-scaling
          - increases pods based on load
     - Self-healing
          - restarts failed containers
     - Load balancing
          - distributes traffic
     - Rolling deployments
          - zero downtime deploy

How do you scale?
     - Using Horizontal Pod Autoscaler based on CPU or custom metrics.

How is traffic handled?
     - Using Services which load balance across pods.

Difference between Kubernetes and Docker?
     - Docker → builds & runs containers
     - Kubernetes → manages containers at scale

What is Ingress:
    - Ingress is an API object that manages external HTTP/HTTPS access to services with routing rules (host/path-based).

How do you deploy a Node.js app on Kubernetes?
     - “First I containerize the app using Docker, then create a Deployment YAML to define replicas, and expose it using a Service.”



What is CI/CD?
CI (Continuous Integration)
     - Developers merge code frequently
     - Automated tests run on every merge
     - Catches bugs early
CD (Continuous Delivery/Deployment)
     - Automates release process
     - Deploy to staging/production automatically
     - Faster, safer releases
Why important?
     - Faster delivery
     - Fewer bugs
     - Reliable releases

What are liveness and readiness probes?
     - Liveness probe → checks if app is alive (restart if dead)
     - Readiness probe → checks if app is ready to receive traffic
     - “They help Kubernetes decide when to restart a pod or when to send traffic to it.”

What is rolling deployment?
     - “It updates pods gradually without downtime by replacing old pods with new ones step by step.”


What is a namespace in Kubernetes?
     - “A namespace is a way to divide cluster resources between multiple users or teams.”

How do you handle persistent storage in Kubernetes?
     - Using Persistent Volumes (PV) and Persistent Volume Claims (PVC).

How do you secure a Kubernetes cluster?
     - RBAC, Network Policies, Secrets, and TLS certificates.

What is a ConfigMap and Secret in Kubernetes?
     - ConfigMap → non-sensitive config data
     - Secret → sensitive data (passwords, API keys)

Explain horizontal vs vertical scaling in Kubernetes.
     - Horizontal → add more pods
     - Vertical → increase resources (CPU/RAM) per pod

How do you debug a failing pod?
     - “I check logs using kubectl logs, inspect pod status with kubectl describe pod, and verify events like crashes or probe failures.”
