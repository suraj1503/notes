Json Web Token:
    - A JSON Web Token (JWT) is a compact, URL-safe means of representing claims to be transferred between two parties. 
        Each JWT contains an encoded header, payload, and signature.
    - Structure: [Base64(Header)][.][Base64(Payload)][.][Signature] 
        1. Header: { "alg": "HS256", "typ": "JWT" }
            - The header typically consists of two parts — the token type (JWT) and the signing algorithm being used, 
                    such as HMAC SHA256 or RSA. 
        2. Payload: { "id": 1, "name": "John Doe", "iat": 1715402838 }
            - The payload contains the "claims", which is information about the user and other data.
                Claims can be of three types: 
                    * Registered Claims: Claims that are predefined and have no commercial value. Examples: iss, exp, sub, aud, iat, etc.
                    * Public Claims: Claims that are registered by the JWT, but are not standardized. Examples: user_id, role, etc.
                    * Private Claims: Claims that are custom-defined by the users and have no commercial value. Examples: user_id, role, etc.
        3. Signature: <HMACSHA256_SIGNATURE>
            - To create the signature part, you need to take the encoded header, encoded payload, a secret, and the algorithm 
                specified in the header, then sign that with the secret. The signature is used to verify that the sender 
                of the JWT is who it says it is and to ensure that the message wasn’t changed along the way.
    
    - Advanages of JWT:
        * Stateless: JWTs store information in the token itself, so the server doesn’t need to store any session data.
        * Compact: JWTs are small and can be easily transmitted in URLs, query parameters, or HTTP headers.
        * Scalable: JWTs can be used in distributed systems, allowing for horizontal scaling of authentication.
        * Secure: JWTs are signed using cryptography, ensuring that the token hasn’t been tampered with.
        * Flexible: JWTs can be used to store custom claims, allowing for flexible authentication and authorization.
        * Mobile and API friendly: Ideal for mobile apps and APIs where state management can be challenging.
        * Scalability: Supports cross-domain authentication and simplifies load balancing in distributed systems.
    
    - Disadvanages of JWT:
        * Storage of sensitive information is not recommended as JWT is Base64 encoded and can be easily decoded.
        * JWT is stateless, so we cannot revoke the token before its expiry.
        * JWT can be larger than sessions as it contains claims.
        * JWT is vulnerable to replay attacks if not properly secured.

    - Best Practices for JWT Implementation:
        * Secure Storage: Store JWTs in HTTP-only cookies to prevent access from JavaScript, reducing the risk of XSS attacks.
        * Token Expiration: Set a reasonable expiration time on JWTs to limit the time window for potential misuse.
        * Token Revocation: Have a mechanism to revoke or blacklist compromised tokens to enhance security.
        * Use HTTPS: Ensure that all communications between the client and server use HTTPS to prevent eavesdropping and man-in-the-middle attacks.
        * Don’t Store Sensitive Data: Avoid storing sensitive data in the JWT payload, as the payload is easily readable once base64-decoded.
 
Difference between Session and JWT:
    | Concept                              | Session (Stateful)                                                | Token / JWT (Stateless)                                                                               |
    | ------------------------------------ | ------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
    | Who stores data?                 | Server (memory/Redis/DB)                                          | Client (token itself holds data)                                                                      |
    | What client has?                 | `sessionId` (just a reference)                                    | Full identity inside token (payload)                                                                  |
    | Frontend request (how it’s sent) | Browser automatically sends cookie:<br>`Cookie: sessionId=abc123` | Usually sent manually:<br>`Authorization: Bearer <token>`<br>OR via cookie:<br>`Cookie: token=abc123` |
    | Backend processing               | Looks up session store → fetch user                               | Verifies token signature → extracts user                                                              |
    | DB lookup needed?                | ✅ Yes (to fetch session/user)                                     | No (unless extra validation needed)                                                                 |
    | State management                 | Stateful (server remembers users)                                 | Stateless (server doesn’t store user state)                                                           |
    | Scalability                      | ❌ Hard (needs shared store like Redis)                            | ✅ Easy (no shared state needed)                                                                       |
    | Performance                      | Slightly slower (DB/cache lookup)                                 | Faster (just verify token)                                                                            |
    | Logout                           | ✅ Easy (delete session from server)                               | ❌ Hard (token valid until expiry)                                                                     |
    | Revocation                       | Immediate (delete session)                                        | Difficult (needs blacklist/short expiry)                                                              |
    | Security (data exposure)        | Safe (data stays on server)                                       | Payload visible (but signed, not encrypted)                                                           |
    | Token size                       | Small (just ID)                                                   | Larger (payload + signature)                                                                          |
    | Storage on frontend              | Cookie (automatic)                                                | localStorage / cookie / memory                                                                        |
    | CSRF risk                        | ⚠️ Yes (cookies auto-sent)                                        | ⚠️ Yes if stored in cookies<br>❌ No if in localStorage                                                |
    | XSS risk                         | Low (httpOnly cookie)                                             | High if stored in localStorage                                                                        |
    | Mobile/API friendly              | ❌ Less ideal                                                      | ✅ Best suited                                                                                         |
    | Microservices                    | ❌ Hard (shared session needed)                                    | ✅ Ideal (token passed between services)                                                               |
    | Load balancer                    | Needs sticky sessions OR shared store                             | No special handling needed                                                                            |
    | Expiration control               | Server-controlled                                                 | Embedded in token (`exp`)                                                                             |
    | Flexibility                      | Less flexible                                                     | Highly flexible (custom payload)                                                                      |

    - Request Flow (Very Important):
        - Session-Based Flow:
            1. Login → server creates session
            2. Server → sends cookie (sessionId)
            3. Browser stores cookie
            4. Every request:
                Browser → automatically sends:
                Cookie: sessionId=abc123
            5. Server:
                → finds session in Redis/DB
                → gets user

        - Token-Based Flow:
            Case 1: Token in Header (most common in APIs)
                1. Login → server returns token
                2. Frontend stores token (memory/localStorage)
                3. Every request:
                    Authorization: Bearer <token>
                4. Server:
                    → verifies token
                    → extracts user

            Case 2: Token in Cookie (like your setup)
                1. Login → server sets cookie:
                    Cookie: token=abc123
                2. Browser stores it
                3. Every request:
                    Cookie: token=abc123 (auto)
                4. Server:
                    → verifies token

        - Modern apps use: JWT (short-lived) + Refresh Token + HttpOnly Cookie + CSRF

    - JWT+Refresh Token Flow:
        1. Login → access + refresh tokens set in cookies
        2. Browser stores both automatically
        3. API request → accessToken sent automatically
        4. Server verifies accessToken
            → If valid → success
        5. If expired:
            → Server returns 401
        6. Frontend interceptor catches 401
            → Calls /refresh (endpoint)
        7. RefreshToken sent automatically via cookie
        8. Server verifies refreshToken
            → Issues new accessToken
        9. Browser updates cookie
        10. Frontend retries original request → success

CSRF (Cross-Site Request Forgery): 
    - An attacker tricks a logged-in user’s browser into making a request without their consent
    - CSRF Attack Flow (Step-by-Step):
        1. User logs into bank.com
        2. Browser stores cookie (sessionId)
        3. User visits evil.com
        4. evil.com silently sends request:
        POST bank.com/transfer
        5. Browser automatically attaches cookie:
        Cookie: sessionId=abc123
        6. Server thinks:
        "Valid user → allow request"
        7. Money transferred
    Note: Why this works? Because: Browser auto-sends cookies and server trusts cookie blindly
    - CSRF Protection Flow (How We Fix It):
        1. User logs in
            → gets cookie (session/JWT)
        2. Frontend requests CSRF token:
            GET /csrf-token
        3. Server sends:
            csrfToken = random123
        4. Frontend stores token (memory)
        5. User performs action:
            POST /transfer
        6. Request contains:
            Cookie: sessionId=abc123
            CSRF-Token: random123
        7. Server checks:
            ✔ Cookie valid
            ✔ CSRF token matches
        8. Request allowed ✅
    - One liner flow: Login → Get Cookie → Get CSRF Token → Send both → Server validates

Difference between encoding, encryption and hashing:
    #Encoding:
        - Encoding is the process of converting data from one format to another so it can be easily transmitted or stored.
        - Key idea
            * NOT for security ❌
            * Anyone can decode it ✅
            * No secret key required
        - Example: Base64 encoding
            const encoded = Buffer.from("hello").toString("base64");
            console.log(encoded); // aGVsbG8=

            const decoded = Buffer.from(encoded, "base64").toString("utf-8");
            console.log(decoded); // hello
        - Real-world use:
            * Sending binary data in APIs (images in JSON)
            * Email attachments
            * URL encoding
    
    #Encryption:
        - Encryption converts readable data (plaintext) into unreadable data (ciphertext) using a key, so only authorized parties can read it.
        - Key idea:
            * Used for security
            * Requires a key
            * Reversible (with correct key)
        - Types:
            * Symmetric encryption (same key)
                Example: AES
            * Asymmetric encryption (public + private key)
                Example: RSA
        - Code Snippet:
            const crypto = require("crypto");
            const algorithm = "aes-256-cbc";
            const key = crypto.randomBytes(32);
            const iv = crypto.randomBytes(16);

            // Encrypt
            const cipher = crypto.createCipheriv(algorithm, key, iv);
            let encrypted = cipher.update("hello", "utf8", "hex");
            encrypted += cipher.final("hex");
            console.log("Encrypted:", encrypted);

            // Decrypt
            const decipher = crypto.createDecipheriv(algorithm, key, iv);
            let decrypted = decipher.update(encrypted, "hex", "utf8");
            decrypted += decipher.final("utf8");
            console.log("Decrypted:", decrypted);
        - Real-world use
            * HTTPS (SSL/TLS)
            * Secure messaging
            * Protecting sensitive data (like API tokens)

    #Hashing:
        - Hashing converts data into a fixed-length string (hash) using a mathematical function.
        - Key idea:
            * One-way ❌ (cannot reverse)
            * Deterministic (same input → same output)
            * No key required (usually)
        - Examples:
            * SHA-256
            * bcrypt
        - Code Snippet:
            const crypto = require("crypto");
            const hash = crypto.createHash("sha256").update("password123").digest("hex");
            console.log(hash);
        - Real-world use: 
            * Password storage 🔥 (VERY IMPORTANT)
            * Data integrity verification
            * Digital signatures

    #Key Differences (Interview Gold Table):
        | Feature      | Encoding        | Encryption       | Hashing                         |
        | ------------ | --------------- | ---------------- | ------------------------------- |
        | Purpose      | Data formatting | Data security    | Data integrity / password store |
        | Reversible   | ✅ Yes           | ✅ Yes (with key) | ❌ No                            |
        | Key required | ❌ No            | ✅ Yes            | ❌ No (mostly)                   |
        | Security     | ❌ None          | ✅ High           | ✅ High (one-way)                |
        | Output       | Readable format | Ciphertext       | Fixed-length hash               |
        | Example      | Base64          | AES, RSA         | SHA-256, bcrypt                 |

All about HTTP/HTTPS(Hyper Text Transfer Protocol/Secure):
    #HTTP:
        - HTTP is an application-layer(Layer 7) protocol used for communication between client (browser/app) and server.
        - Runs on top of TCP, It follows: Request → Response model
        - HTTP Flowchart:
        ┌──────────────┐
        │  Client      │
        │ (Browser/App)│
        └──────┬───────┘
               │
               │ 1. DNS Lookup (domain → IP)
               ▼
        ┌──────────────┐
        │   Server IP  │
        └──────┬───────┘
               │
               │ 2. TCP 3-Way Handshake
               │    SYN → SYN-ACK → ACK
               ▼
        ┌──────────────┐
        │ TCP Connection│
        └──────┬───────┘
               │
               │ 3. HTTP Request (plain text)
               │    GET / POST / etc.
               ▼
        ┌──────────────┐
        │ Web Server   │
        │ (NGINX/Node) │
        └──────┬───────┘
               │
               │ 4. Process request
               │    (DB / Logic)
               ▼
        ┌──────────────┐
        │ HTTP Response │
        │ (200/404 etc)│
        └──────┬───────┘
               │
               │ 5. Response sent (plain text)
               ▼
        ┌──────────────┐
        │   Client      │
        │ Renders page  │
        └──────────────┘

        - HTTPS Flowchart:

Polling, Long Polling and Websockets:
    # Polling:
        - Client keeps asking the server again and again at fixed intervals: “Do you have new data?”
        - How it works:
            1. Client sends request to server
            2. Server responds immediately (with data or empty)
            3. Client waits for some time (e.g., 2 sec)
            4. Client sends request again
        - Example: (frontend code)
            setInterval(async () => {
                const res = await fetch('/messages');
                const data = await res.json();
                console.log(data);
            }, 2000);
        - Key Points:
            * Simple to implement
            * Wasteful if no new data
            * High latency (depends on interval)
            * Http overhead
        - Problem: 
            * If data comes at 1.1 sec but interval is 2 sec → user sees delay
        - Use Polling when:
            * Simple app
            * Data changes rarely
            * Real-time not important

    # Long Polling:
        - Client asks once, but server waits until data is available before responding. “Tell me when something happens.”
        - How it works:
            1. Client sends request
            2. Server does NOT respond immediately
            3. Server waits until:
                -> new data arrives OR
                -> timeout occurs
            4. Server responds
            5. Client immediately sends next request
        - Example:
            async function poll() {
                const res = await fetch('/getUpdates');
                const data = await res.json();
                console.log(data);

                // Immediately request again
                poll();
            }
            poll();
        - Key Points:
            * Reduces unnecessary requests
            * Near real-time
            * Still HTTP-based
        - Problem:
            * Each response closes connection → new request needed
            * Still overhead of HTTP headers again & again
        - Use Long Polling when:
            * Need near real-time
            * Cannot use WebSockets (firewalls, legacy infra)
    
    # WebSockets:
        - Create a persistent, full-duplex connection between client and server. “Let’s keep the connection open and talk anytime.”
        - How it works:
            1. Client sends HTTP request (Upgrade request)
            2. Server upgrades to WebSocket
            3. Connection stays open
            4. Both client & server can send data anytime
        - Example (Node.js + WebSocket):
            * Server:
                const WebSocket = require('ws');
                const wss = new WebSocket.Server({ port: 3000 });
                wss.on('connection', (ws) => {
                    ws.send('Connected!');
                    ws.on('message', (msg) => {
                        console.log('Received:', msg.toString());
                    });
                });
            * Client:
                const socket = new WebSocket('ws://localhost:3000');
                socket.onmessage = (event) => {
                    console.log(event.data);
                };
                socket.send("Hello server");
        - Key Points:
            * Real-time communication
            * Bi-directional
            * Low latency
            * Efficient after connection established
        - Problem:
            * More complex
            * Needs connection management (reconnect, scaling)
            * Not ideal for simple apps
        - Use WebSockets when:
            * Chat apps (like your SupX 🔥)
            * Live notifications
            * Multiplayer games
            * Stock price updates
        
    # Difference:   
        | Feature              | Polling                  | Long Polling            | WebSockets              |
        | -------------------- | ------------------------ | ----------------------- | ----------------------- |
        | Connection Type      | Short-lived HTTP         | Long-held HTTP          | Persistent TCP          |
        | Who initiates        | Client only              | Client                  | Both client & server    |
        | Latency              | High                     | Medium (near real-time) | Low (real-time)         |
        | Server Load          | High (many requests)     | Medium                  | Low (after connection)  |
        | Bandwidth Usage      | High                     | Medium                  | Low                     |
        | Real-time capability | ❌ Poor                   | ⚠️ Moderate             | ✅ Excellent             |
        | Complexity           | Easy                     | Medium                  | Hard                    |
        | Overhead             | High (headers each time) | Medium                  | Low                     |
        | Use case             | Simple apps              | Chat (older systems)    | Chat, live apps, gaming |
